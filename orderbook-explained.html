1What this thing is呢個 project 係乜嘢

It's the beating heart of a stock/crypto exchange, in ~270 lines of Go with zero dependencies. When you place an order on Coinbase or NASDAQ, somewhere a piece of code decides who trades with whom, at what price, in what order. That piece is a matching engine sitting on top of an order book. This repo is a clean reference implementation of exactly that, for a single instrument (one symbol, like just "BTC").

It's a portfolio project, so it's optimised for clarity and correctness, not raw speed — and the README is honest about that.

簡單講,佢就係證券/加密貨幣交易所嘅心臟,用 Go 寫,得返大概 270 行,而且冇用任何第三方 library(zero dependency)。你喺 Coinbase 或者證券行落單嗰陣,後台一定有段 code 決定:邊個同邊個成交、用咩價成交、邊個排頭先成交。呢段 code 就叫做 matching engine(撮合引擎),佢搭喺一個 order book(訂單簿)上面。呢個 repo 就係將呢樣嘢乾淨咁實現出嚟,專做一隻商品。

因為佢係 portfolio project(作品集),所以重點係清晰同正確,唔係追求極速。README 自己都好坦白咁講明呢點。

2The domain concepts行業概念(好多人會跳過)

An order book is just the list of everyone who wants to trade but hasn't yet. Two sides:

  • Bids = buyers. ("I'll buy 5 at price 99.")
  • Asks (offers) = sellers. ("I'll sell 5 at price 101.")

A resting order = an order sitting on the book waiting. A maker makes liquidity (posted and waited); a taker takes liquidity (their incoming order matches a resting one). The same order can be a taker for part and a maker for the rest.

The spread = the gap between best bid and best ask. A trade only happens when an incoming order crosses the spread (a buyer willing to pay ≥ the lowest ask, or a seller willing to accept ≤ the highest bid).

Price-time priority — the whole point

  1. Price first. Best price wins. A taker buying fills the cheapest ask first; a taker selling fills the highest bid first.
  2. Time second (FIFO). Tie on price? Whoever rested earliest fills first. You can't jump the queue by being louder — only by offering a better price or arriving earlier.
Maker-price execution (the subtle rule): a trade executes at the resting (maker) order's price, not the taker's. Put in a buy limit "willing to pay up to 105" against a resting ask at 100 → you pay 100. That's price improvement, and it's how real markets work.

Limit vs market

  • Limit = "trade, but only at my price or better." Unfilled remainder rests on the book (becomes a maker).
  • Market = "trade NOW at any price." Sweeps the book greedily; any unfilled remainder is dropped — a market order never rests.
Integer prices. Everything is int64 (cents / lots), never float64. Floats can't represent 0.10 exactly, so financial code uses integers for exact arithmetic. A real correctness decision, not a toy choice.

Order book(訂單簿)其實就係一張清單,記住晒所有「想買賣但未成交」嘅人。分兩邊:

  • Bids(買盤) = 啲買家。(「我出 99 蚊買 5 個。」)
  • Asks(賣盤) = 啲賣家。(「我出 101 蚊賣 5 個。」)

Resting order(掛單) = 一張喺度排緊隊、等緊成交嘅單。Maker(掛單者)製造流動性(早早掛咗單喺度等);Taker(食單者)攞走流動性(新單一入嚟就同人哋啲掛單成交)。同一張單可以一部分做 taker,剩低嗰部分做 maker。

Spread(價差) = 最佳買價同最佳賣價之間嘅罅。只有當新單「越過」呢條罅(cross the spread)先會成交 —— 即係買家肯出 ≥ 最低賣價,或者賣家肯收 ≤ 最高買價。

價格-時間優先 —— 成個 project 嘅靈魂

  1. 價格行先。最好價贏。Taker 買嘢,最平嗰張 ask 先成交;Taker 賣嘢,最貴嗰張 bid 先成交。
  2. 時間行第二(FIFO)。價錢打和?咁就邊個早掛邊個先成交。你冇得靠惡、靠嗌交插隊,你只可以靠出更好嘅價或者早啲到。
Maker 價成交(最微妙嗰條規則):成交價係跟掛單者(maker)嘅價,唔係跟 taker 嘅價。你掛張買單話「我最多肯畀 105」,但市場有張 ask 掛喺 100 → 你最後係畀 100。呢個叫做 price improvement(價格改善),真實市場就係咁運作。

限價單(Limit)同市價單(Market)

  • Limit 限價單 = 「要成交,但一定要喺我嗰個價或者更好。」食唔晒嘅就掛喺 book 度等(變咗 maker)。
  • Market 市價單 = 「即刻成交,咩價都得。」貪心咁掃晒個 book;食唔晒嗰部分會直接掉咗(drop) —— 市價單永遠唔會掛喺度。
整數價。全部嘢都用 int64(分/手),永遠唔用 float64。因為 float 連 0.10 都冇辦法準確表示,所以金融 code 一定用整數嚟做精確運算。呢個係真實世界嘅正確性決定,唔係求其。

internal/orderbook/orderbook_test.go # 12 tests, one per rule

internal/ is a Go convention: nothing outside this module can import that package. It enforces "this is the engine — talk to it only through its public API." The engine does no printing, no networking, no concurrency — that separation is deliberate and is what makes it testable.

internal/ 係 Go 嘅慣例:呢個 module 以外嘅嘢冇權 import 入面嘅 package。咁就強制咗「呢個係引擎,你只可以透過佢嘅公開 API 同佢溝通」。引擎本身冇 print、冇網絡、冇 concurrency —— 呢個分離係刻意嘅,亦都係佢咁易 test 嘅原因。

4Data structures — the clever part數據結構 —— 最精彩嘅部分

The challenge: answer "what's the best price?" instantly, fill orders in time order, AND cancel any order by ID instantly. No single structure does all three, so the code layers a few:

挑戰係:要即刻答到「最佳價係幾多?」,又要跟時間順序成交,又要可以靠 ID 即刻 cancel 任何一張單。冇單一個結構做得晒呢三樣,所以段 code 疊咗幾層落去:

OrderBook
 ├─ bids  *side ─┐
 ├─ asks  *side ─┤  // each side:
 │               │    levels  map[price]*level   → O(1) jump to a price
 │               │    prices  []int64 (sorted)   → find the best price
 │               └────  isBid  bool
 └─ resting map[string]*resting   → O(1) find ANY order by ID (for cancel)

Each level (all orders at one price) holds its orders in a container/list — a doubly-linked list used as a FIFO queue:

每個 level(同一個價位嘅所有單)用 container/list —— 即係一條雙向鏈表,當做 FIFO 排隊:

level @ price 100:  [a1] ⇄ [a2] ⇄ [a3]
                     ↑ front (fills first)   ↑ back (newest)
The elegant trick: removing from the middle (a cancel) is O(1) if you already hold a pointer to the node — and the engine does. Each resting struct stores its own elem, level and side back-pointers. So Cancel = look up ID in the map → follow the pointer → unlink → done, all O(1).
最靚仔嘅技巧:如果你已經揸住嗰個 node 嘅指針,喺中間刪一張單(cancel)就係 O(1) —— 而引擎正正揸住。每個 resting struct 自己存住 elem、level、side 嘅反向指針。所以 Cancel = 用 ID 喺 map 度搵到 → 跟住指針 → 解開個 link → 搞掂,全部 O(1)。

The prices slice stays sorted ascending always, so: best ask = prices[0] (lowest sell), best bid = prices[len-1] (highest buy). That's the whole best() function.

prices 個 slice 永遠由細到大排好序,所以:最佳 ask = prices[0](最低賣價),最佳 bid = prices[len-1](最高買價)。成個 best() function 就係咁簡單。

5The matching algorithm撮合演算法

The whole engine is one function: Submit. Its shape:

成個引擎其實就係一個 function:Submit。佢個骨架:

validate (qty > 0, limit price > 0, no duplicate ID)
pick the OPPOSITE side to match against
for remaining > 0:
    best, ok := opposite.best()            // peek the best price
    if no opposite side → stop
    if LIMIT and best price is worse than my limit → stop   ← key line
    walk this price level FIFO:
        maker = front of the queue
        q = min(my remaining, maker's remaining)
        record a Trade at the MAKER's price
        decrement both; if maker hits 0, remove + delete from id map
    drop the level if it's now empty
after the loop: if LIMIT and remainder > 0 → rest it on MY side
驗證 (數量 > 0、限價單嘅價 > 0、ID 唔可以重複)
揀「對面」嗰邊嚟撮合
當 我仲有貨未食完:
    best, ok := 對面.best()                // 偷睇對面最佳價
    如果 對面冇單 → 收手
    如果 係 LIMIT 而 best 價已經差過我嘅限價 → 收手   ← 關鍵嗰行
    跟住 FIFO 行呢個價位嘅隊:
        maker = 隊頭嗰張
        q = min(我剩低嘅, maker 剩低嘅)
        記低一單 Trade,用 MAKER 嘅價
        兩邊都扣數;maker 扣到 0 就刪走 + 喺 id map 度 delete
    呢個價位空咗就 drop 佢
loop 完:如果係 LIMIT 而仲有剩 → 將佢掛喺「我自己」嗰邊
One line distinguishes limit from market: the price check. A market order skips it, so it keeps sweeping regardless of price; a limit order breaks the moment the best opposite price crosses past its limit.
分辨限價同市價嘅就係嗰一行價格檢查。市價單完全跳過佢,所以唔理價錢一路掃落去;限價單就喺對面最佳價一越過佢個限價嗰刻即刻 break。

7Honest trade-offs老實嘅工程取捨

  • Not concurrency-safe. A real venue runs one matching thread per instrument fed by a channel — no locks needed. Listed as next step #1.
  • Inserting a price level is O(n) — the sorted slice shifts elements. Fine for clarity; production uses a tick-indexed array or an intrusive tree.
  • The test suite is the real spec. 12 tests, one per rule. To learn the engine, read orderbook_test.go top to bottom — each test is a tiny worked example.
  • 唔係 concurrency-safe。真實交易所係一隻商品一條撮合 thread,靠 channel 餵單入去 —— 咁就唔使用鎖。列咗做頭號 next step。
  • 插入一個新價位係 O(n) —— 排好序嘅 slice 要 shift 啲元素。為咗清晰冇問題;production 會用以 tick 做 index 嘅 array 或者 intrusive tree。
  • 測試套件先係真正嘅 spec。12 個 test,一條規則一個。想真正學識個引擎,由頭到尾讀 orderbook_test.go —— 每個 test 就係某條規則嘅迷你實例。

8Concurrency — do you need a mutex?並發 —— 使唔使 mutex 鎖?

In reality thousands of orders hit at the same moment. So do you need a sync.Mutex? Surprisingly, real high-performance exchanges usually don't. Here's why.

現實中一定有幾千張單同一刻殺到。咁係咪要用 sync.Mutex 鎖?可能同你諗嘅相反:真實嘅高頻交易所通常唔用。等我解釋。

1 · Matching is inherently serial1 · 撮合本質上一定要排隊

If two buys both try to take the same resting ask (which only has 5) at the same instant, they're racing for the same liquidity:

如果兩張買單同一刻都想食市場上同一張 ask(嗰張得 5 個),佢哋係爭緊同一批 liquidity:

        ASK 101  x5   ← 得 5 個 / only 5 available
       ↗        ↖
  BUY 5      BUY 5     ← 兩張同時殺到 / both arrive at once
Run them truly in parallel and both might think they got 5 → you sold 10 of something that only had 5 → double-spend. So matching must be serialised no matter what. The only question is how you serialise.
如果真係並行處理,兩張可能都以為自己食到 5 → 賣咗 10 個但其實得 5 個 → 超賣(double-spend)。所以撮合無論點都一定要序列化。問題淨係:你用咩方法去排隊。

Option A · A mutex (works, but not best)方法 A · 用 Mutex(得,但唔係最好)

You can wrap every Submit/Cancel in a lock:

你係可以喺每次 Submit/Cancel 外面包個鎖:

func (ob *OrderBook) Submit(o Order) ([]Trade, error) {
    ob.mu.Lock()
    defer ob.mu.Unlock()
    // ... matching logic ...
}

Correct and simple at small scale. But under load: lock contention (hundreds of threads block waiting), unpredictable tail latency (you never know when you'll get the lock — fatal for trading), and it buys no real parallelism anyway — the critical section is still one-at-a-time because of the double-spend problem above.

細規模冇問題。但高負載下:爭鎖(lock contention)(幾百條 thread 喺度等),延遲忽快忽慢(唔知幾時搶到鎖 —— 交易系統最忌),而且根本攞唔到真正並行 —— 因為上面講嘅超賣問題,撮合嗰段始終一次一張。

Option B · One matching thread + a queue (no lock!)方法 B · 單一撮合 thread + 隊列(冇鎖!)

The industry standard (LMAX Disruptor, most real exchanges): one matching thread per instrument.

業界標準(LMAX Disruptor、大部分真實交易所):一隻商品一條撮合 thread。

  many producers (intake)              one consumer (matching)
  goroutine 1 ─┐
  goroutine 2 ─┤──→  [ order channel / queue ]  ──→  match goroutine ──→ mutate book
  goroutine 3 ─┘         (orders line up)            (the ONLY one touching the book)
type request struct {
    order Order
    reply chan []Trade   // result comes back over a channel
}

// Only THIS one goroutine ever touches the OrderBook.
func (ob *OrderBook) Run(in <-chan request) {
    for req := range in {              // the channel does the queuing
        trades, _ := ob.Submit(req.order)
        req.reply <- trades
    }
}
The key: only one goroutine ever touches the book → there is no race → no lock needed. The concurrency moved to the channel: many producers push, one consumer pops. The channel handles thread-safety; the book stays blissfully single-threaded.
重點:成個 book 由頭到尾淨係一條 goroutine 掂 → 根本冇得 race → 唔使鎖。並發性搬咗去個 channel 度:好多 goroutine 塞單入去,一條攞出嚟做。個 channel 處理咗 thread-safety,個 book 完全唔使理。

Why it beats a mutex點解好過 Mutex

MutexOne thread + queue單 thread + 隊列
Latency延遲 jittery (depends on lock)忽快忽慢(睇搶唔搶到鎖) stable, predictable穩定、可預測
Cache cores bounce the data around幾個 core 搶同一份 data book stays hot on one corebook 長留同一 core 嘅 cache
Replayable可重播 order non-deterministic次序唔確定 deterministic → fully replayable次序確定 → 可完全重播
"Replayable" matters a lot: a deterministic order means you can re-run the exact same order stream and get the exact same result — essential for debugging, audits and regulatory reporting. A mutex's "whoever grabs it first" gives you no such guarantee.
「可重播」好緊要:次序確定即係你可以將同一串訂單一模一樣咁重跑一次,得出一樣嘅結果 —— 查 bug、審計、監管報告全靠呢樣。Mutex 嗰種「邊個搶到算邊個」就冇呢個保證。

So how do you use all your CPU cores? Shard by symbol咁點食盡多核 CPU?Shard by symbol

  BTC  book  →  match thread 1   ┐
  ETH  book  →  match thread 2   ├─ truly parallel, different cores
  AAPL book  →  match thread 3   ┘   no shared state → still no locks

BTC and ETH books are unrelated (a BTC buy never matches an ETH sell), so each runs on its own core in true parallel — and still needs no lock. You get parallelism across instruments, not within one book (which can't be parallelised anyway).

BTC 同 ETH 嘅 book 完全冇關係(BTC 買單唔會同 ETH 賣單成交),所以各自喺唔同 core 真正並行 —— 而且仍然唔使鎖。並行性係喺「商品之間」攞,唔係喺「同一個 book 入面」攞(嗰度本身就並行唔到)。

One-line summary: matching must be serial, so the question isn't "lock or not" but "how to serialise." Real systems pick "one thread draining one queue" — no lock, stable latency, replayable. Scale with one thread per symbol. This repo doesn't do it yet — it's next step #1 in the README (see the comment at orderbook.go:130-131).
一句總結:撮合一定要排隊,所以重點唔係「使唔使鎖」,而係「點排隊」。真實系統揀「一條 thread 食一條隊列」—— 冇鎖、延遲穩定、可重播。要食多核就一隻商品一條 thread。呢個 repo 而家未做,係 README 嘅頭號 next step(睇 orderbook.go:130-131 個註解)。
